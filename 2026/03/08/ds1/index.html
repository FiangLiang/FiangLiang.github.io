Lec 1: Introduction

第一章照例还是讲一些宽泛的概念。我尽量挑选重点内容来说明。

1 数据库系统应用

数据库管理系统 (DataBase-Management System, DBMS) 由一个互相关联的数据的集合和一组用以访问这些数据的程序组成。 这个数据集合通常称作数据库 (database)。

概括地说，有两种使用数据库的方式：

  • 联机事务处理，即大量的用户使用数据库，每个用户检索相对少量的数据，进行小的更新。这是数据库应用系统的绝大多数用户的主要使用方式。
  • 数据分析，即审阅数据，给出结论，并推导出规则或决策程序，以用于驱动业务决策。

2 数据库系统的目标

在早期，数据库应用是建立在文件处理系统之上的，但这样有以下问题:

  • 数据的冗余和不一致性
    • 数据冗余是指相同的信息可能在几个地方 （文件）重复存储
    • 数据冗余可能导致数据的不一致性，即同一数据的不同副本不一致
  • 数据访问困难：无法满足个性化的数据查询需求
  • 数据隔离：数据可能散落在不同的文件内，且文件可能具备不同的格式
  • 完整性问题：存储在数据库的数据值必须满足特定类型的一致性限制，但是文件系统很难维护这样的限制
  • 原子性问题：在文件系统中，由于数据分散，数据的原子性难以保证
  • 并发访问异常：多个用户同时访问数据库时，由于数据分散，数据的并发访问难以保证
  • 安全性问题：不同的用户可能有不同的权限，在文件系统中，由于数据分散，数据的安全性难以保证

在这样的情形下，数据库系统应运而生

3 数据视图

数据库系统的一个主要目的是给用户提供数据的抽象视图。

3.1 数据模型

数据库结构的基础是数据模型——一个描述数据、数据联系、数据语义以及一致性约束的概念工具的集合

数据模型可被划分为四类：

  • 关系模型（relational model）

    用表的集合来表示数据和数据间的联系

  • 实体-联系模型（entity-relational/E-R model）

    利用实体、属性和实体间的关系来描述现实世界中的数据

  • 半结构化数据模型（semi-structured data model）

    主要用于描述部分有组织但不严格依赖固定模式的数据

  • 基于对象的数据模型（object-based data model）

    结合面向对象编程思想，把对象、类、方法等引入数据管理中

3.2 数据抽象

抽象(abstraction) 这一概念在数据库管理中非常重要，它能让我们在不了解设备或系统的底层细节的情况下，能够使用这些复杂的设备或系统。

开发者通过以下几层数据抽象(data abstraction) 来简化用户与数据库系统的交互：

  • 物理层（physical level）

    数据库的最低层次，描述了数据在磁盘上的存储方式

  • 逻辑层（logical level）

    数据库的中间层次，描述了数据的逻辑结构，即数据库中存在什么数据，以及数据之间的联系。

  • 视图层（view level）

    数据库的最高层次，描述了用户所见到的数据，数据库系统存在多个视图，每个用户可以看到不同的视图。

数据独立性是指数据库这三个层次之间的独立性：

  • 物理数据独立性

    数据库的物理层与逻辑层之间的独立性，即数据库的逻辑层不依赖于数据库的物理层，当修改物理层时，无需修改逻辑层使之适配

  • 逻辑数据独立性

    数据库的逻辑层与视图层之间的独立性，即数据库的视图层不依赖于数据库的逻辑层，当修改逻辑层时，无需修改视图层使之适配

3.3 实例与模式

模式与实例的关系类似于变量声明与变量值的关系：

  • 模式（Schema）：描述了数据库的逻辑结构，即数据库中存在什么数据，以及数据之间的联系
  • 实例（Instance）：描述了数据库中的具体数据

按不同的抽象层次划分，数据库系统也有三种模式：

  • 物理模式：描述在物理层上的数据库设计
  • 逻辑模式：描述在逻辑层上的数据库设计
  • 子模式：描述了数据库的不同视图

4 数据库语言

数据库语言主要分为数据定义语言和数据操纵语言两种，它们共同构成了数据库系统的语言系统（例如SQL）

4.1 数据定义语言

  • 该语言还要指明完整性约束（integirty constraints），包括：
    • 域约束（domain constraints）
    • 参照完整性（referential integrity）
    • 权限（authorization）
  • 这些描述数据的语句称为元数据（Metadata），即数据的数据（data about data），它们被存储在一个称为数据字典（data dictionary）的特殊文件中，包括：数据库模式、数据存储结构、访问模式和约束、统计信息、权限

4.2 数据操纵语言

数据操纵语言，Data Manipulation Language (DML)，允许用户访问和操纵那些被合适的数据模型组织过的数据，包括检索、插入、删除、修改等操作。

例如：

DML 可分为以下两种类型：

  • 过程型(procedural) ：用户指明所需数据以及获取数据的方法，比如 C、Pascal、Java 等语言，对应的数学表示为关系代数 (relational algebra)
  • 声明型 (declarative) DMLs：用户只需指明数据，无需指出如何获取数据，相比过程型 DMLs 更易于使用，比如 SQL、Prolog 等语言，对应的数学表示为关系演算 (relational calculus)

4.3 SQL

使用最广泛的查询语言是 SQL（structured query language，结构化查询语言），其特点为：

  • 提供了丰富的 DDL，允许定义带有各种数据类型和完整性约束的表格。
  • 非过程的查询语言，查询将一张或多张表格作为输入，输出一张表格。

像 SQL 这样的非过程型查询语言并不具备与通用图灵机一样强大的能力，因为它无法进行一般编程语言能够做到的计算，也不支持用户输入、输出显示和网络通信等行为。因此这样的计算和行为必须用宿主语言（host language）编写（比如 C/C++、Java、Python 等），并依靠嵌入式的 SQL 查询来访问数据库内的数据。应用程序（application program）就是通过这种方式来和数据库交互的程序。

一般来说，应用程序通过 API (Application Programming Interface) 来向数据库发送 DML 和 DDL 语句并检索数据。下面列出两种常见的 API：

  • ODBC（open database connectivity，开放数据库互连）：定义了 C 和其他语言的 API
  • JDBC（Java database connectivity，Java 数据库互连）：定义了 Java 对应的接口

5 数据库设计

数据库设计步骤如下：

  • 需求分析(requirement analysis)：需要什么数据、应用和操作
  • 概念设计(conceptual design)：用 E-R 模型或类似的高级数据模型，以及一些算法来对数据进行高层级的描述
  • 逻辑设计(logical design)：将高层的概念模式映射到数据库系统所用到的数据模型的实现中
  • 模式改善(schema refinement)：检查关系模式中的冗余和关系异常
  • 物理设计(physical design)：索引 (indexing)、聚类 (clustering) 和数据库调优 (tuning)
  • 创建和初始化数据库、安全设计(create and initialize database & security design)

6 数据库引擎

数据库引擎是数据库系统的核心组件，负责管理数据库的存储、检索、更新等操作。

数据库的主要部件包括：

  • 存储管理器(storage manager)

    • 负责管理数据库的存储，包括数据的存储、检索、更新等操作。

    • 组成部分

      • 授权和完整性管理器(authorization and integrity manager)
      • 事务管理器(transaction manager)：维护一致性和并发事务执行
      • 文件管理器(file manager)：管理硬盘存储空间分配
      • 缓冲区管理器(buffer manager)：将数据从硬盘拿到主存中，并决定将哪些数据放在缓存中

    • 存储管理器实现了如下数据结构：

      • 数据文件：存储数据库自身
      • 数据字典：存储数据库结构的元数据，尤其是数据库的模式
      • 索引：提供对数据项的快速访问

  • 查询处理器(query processor)

    • 负责解析SQL语句，生成查询计划，执行查询计划，返回查询结果

    • 组成部分

      • DDL 解释器(interpreter)：解释 DDL 语句，并将定义记录在数据字典中
      • DML 编译器(compiler)：将用查询语言表述的 DML 语句翻译为由底层指令构成的评估计划 (evaluation plan)，此外还支持查询优化（选择成本最低的评估计划）
      • 查询评估引擎(query evaluation engine)：执行由 DML 编译器生成的底层指令

  • 事务管理组件(transaction management component)

    • 事务(transaction)：用于在数据库应用中执行单个逻辑功能的一组操作
    • 事务的要求——ACID：原子性(atomicity)、一致性(consistency)、隔离性(isolation)、持久性(durability)
    • 事务管理组件的组成部分：
      • 恢复管理器(recovery manager)：通过备份或恢复子系统，来确保数据库在系统或事务故障时，能够保持数据库的一致（正确）状态
      • 并发控制管理器(concurrency-control manager) 用于控制并发事务之间的互动

7 Database and Application Architecture

整体架构可以参考下面这张图：

可以看到，对于不同类型的用户，数据库提供了不同的用户接口类型：

  • 普通 (naive) 用户：通过预先定义的用户接口（比如 web 或移动应用）与数据库系统交互
  • 应用程序员：编写应用程序，通过多种工具开发用户接口
  • 富有经验的用户：并不通过编写程序与系统交互，而是使用数据库查询语言，或者使用数据分析软件等工具来构建请求
  • 数据库管理员(database administrator, DBA)：具备对数据库和程序的中央控制权的特殊用户

使用数据库作为后端的应用架构，可以分为：

  • 两级架构(two-tier architecture)
    • 应用程序位于客户端机器内，通过查询语言语句调用位于服务器上的数据库系统
  • 三级架构(three-tier architecture)
    • 客户端仅作为前端（浏览器或移动应用等），并不包含任何直接的数据库调用，与应用服务器通信
    • 应用服务器可以与数据库系统通信，以访问数据。应用的商业逻辑 (business logic) 也嵌在应用服务器上
    • 相比两级架构具备更好的安全性和性能
