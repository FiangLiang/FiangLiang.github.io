Lecture 3：倒排索引

据说是 ADS 里最简单的一集，且让我试试咸淡

主要参考了 陈越的Mooc，同时参考了Isshiki修’s Notebook和之前提过的讲义

感觉这一节和之前的平衡树一点关系没有，过渡还是太不自然了

Simple Solutions

搜索引擎是如何实现的？为什么输入几个关键词就能得到相关的网页？

Solution 1: 暴力比较

最直观的方法：如果我们尝试将关键词与所有网页信息都比较一遍，会发什么？

字符串比较大家都应该略知一二，最好复杂度是 O(m+n)O(m+n)O(m+n)，谷歌收录了超过10亿的网页，遍历一次的时间消耗太大，故这个方法不太现实。

Solution 2: 建立矩阵

Doc Text
1 Gold silver truck
2 Shipment of gold damaged in a fire
3 Delivery of silver arrived in a silver truck
4 Shipment of gold arrived in a truck

我们将一个网页简化为一段文字，如上图，一共有 4 个网页

我们可以用矩阵表示关键字与网页的归属关系，如下图

1 2 3 4
a 0 1 1 1
arrived 0 0 1 1
damaged 0 1 0 0
delivery 0 0 1 0
fire 0 1 0 0
gold 1 1 0 1
of 0 1 1 1
in 0 1 1 1
shipment 0 1 0 1
silver 1 0 1 0
truck 1 0 1 1

可以发现，每一个关键词都可以用一个二进制数表示其与各网页的关系，比如：truck 可以用 101110111011 表示

如果我们想找到一个同时包含 silver 和 truck 的网页，可以做按位与操作：1010&1011=10101010\&1011=10101010&1011=1010，即可得对应的网页（传递闭包？（雾 ）

这个方法不用逐一比较，而是预处理了关键字与网页的关系，确实更快。但事实上，如果将关键字与网页的关系用简单的矩阵表示的话，这个矩阵会非常的稀疏，也就是会有大量的 0，这会造成空间的浪费。

那也许你就会问了：既然矩阵太浪费空间了，有没有更不浪费空间的数据结构推荐一下？

有的，兄弟，有的，接下来就为你介绍本章的主角。

Structure of the index

相比Solution 2采用的矩阵，**倒排索引（Inverted File Index）**采用链表的形式，在保证效率的同时能节省大量的空间开支。

Definition

Index：索引，用来定位某一文章里特定词组的体系

Inverted file：倒排文件，是一个指针列表，指针指向特定词组所有出现过的位置；之所以叫 Inverted，是因为其通过词组去找文章，而不是通过文章去找词组

Structure

沿用之前的数据，用 Inverted File Index 表示大概长这样

No. Term Times; Documents
1 a <3; 2,3,4>
2 arrived <2; 3,4>
3 damaged <1; 2>
4 delivery <1; 3>
5 fire <1; 2>
6 gold <3; 1,2,4>
7 of <3; 2,3,4>
8 in <3; 2,3,4>
9 shipment <2; 2,4>
10 silver <2; 1,3>
11 truck <3; 1,3,4>

通过上面这个表格很容易找到某个关键词出现的文章位置，如果有多个关键字，可以先比较各个关键词出现次数，从而简化查找耗时

稍加修改，我们也可以在查找文章序号的同时定位关键字在文章中出现的位置

No. Term Times; Documents Words
1 a <3; (2;6),(3;6),(4;6)>
2 arrived <2; (3;4),(4;4)>
3 damaged <1; (2;4)>
4 delivery <1; (3;1)>
5 fire <1; (2;7)>
6 gold <3; (1;1),(2;3),(4;3)>
7 of <3; (2;2),(3;2),(4;2)>
8 in <3; (2;5),(3;5),(4;5)>
9 shipment <2; (2;1),(4;1)>
10 silver <2; (1;2),(3;3,7)>
11 truck <3; (1;3),(3;8),(4;7)>

Token analyzer and stop filter

中文比英文更难处理，为了避免过于深入，下面仅讨论英文场景

将介绍两个常用的技术

Word stemming

词干分析，将单词转换为其词干的技术。比如 Process，processing，processes，processed 都转换为 processed 来储存，从而优化查找和插入过程。

Stop words

停用词，把一些无法作为关键词的词组给排除，比如 a，the，它们频繁出现在各种网页中，难以作为有效信息被检索。

Vocabulary Scanne and Vocabulary Insertor

用什么数据结构去实现 item 的查找和插入呢？

有这样两种选择，一是以 B 树为代表的平衡树，二是哈希

我们都知道，哈希法的查找是常数级的时间复杂度，空间复杂度也是最优的线性复杂度，似乎哈希是最好的选择

可事实表面，如果是单个词的查找，哈希确实是最优解，但如果涉及区间查找（浏览器通常会对给出的关键字进行模糊匹配），平衡树的效率将高于哈希表（我也不知道什么原理，可能是哈希表复杂度的常数比较大）

Distributed indexing

对于一个搜索引擎，它所需要索引的资料是非常庞大的，肯定没法全都放在一个服务器里，所以我们需要分布式的建立索引。建立索引也有两个方法（感觉这章好多对比啊）。

Term-partitioned index

按照词组来存储，比如把 A 打头的词组放在一个服务器，把 B 打头的词组放在一个服务器……

这种方法的查找效率比较高，但是由于所有形式接近的单词都存储在一个地方，容易造成单点故障（我也不知道为什么）

Document-partitioned index

第二种方法通过文档来分布式储存，相比第一种有更强的容灾性能

Dynamic indexing

随着网页的增多，插入等操作的复杂度会不断提高。为了减缓复杂度的提高，我们可以在 Main Index 以外增加一个 auxiliary index，即辅助性索引。相比 Main Index， 辅助性索引规模更小，处理网页的效率也更高，因此我们通常将新的网页插入辅助性索引中，只有在必要时才将网页合并进 Main Index 中。当然，在搜索操作中，我们要同时在两个索引进行查找。

Compression

下面我们介绍一些简单的空间压缩操作

Term compression

在之前我们介绍了停用词等操作，这里再介绍一个思想：词组不是以一个个字符串的形式储存的，因为每个词组长度不同，容易造成空间的浪费。我们将这些词组连成了一个十分长的字符串，转而用另一个数组来储存每个词组首字母在总字符串的位置。

List compression

对于一个词组，我们要记录它出现在那几篇网页中，然后将文章的序号插入对应的 list 中。假如有一个词，它出现在网页的序号为：2, 5, 7,..., 546484, 546487，...，可以发现随着插入的网页越来越多，对应的序号也越来越大，前面的序号用 short 就能表示，后面就可能只能用 long long 了。为了减少空间消耗，我们可以对这个序号序列做一个差分，变为：2, 3, 2,..., 6, 3,...，需要时做个前缀和就可以得到对应网页序号。（其实个人感觉这样会比较麻烦，不过我也不是很懂，不多做评价）

Thresholding

设定阈值，但说人话，就是通过优先级“剪枝”，以此优化时间复杂度。

Document

满足搜索要求的网页可能有很多，但我们不需要把它们全部作为结果返回。我们可以设定一个阈值 x，即只返回优先级最高的前 x 篇网页。

Query

一次搜索中给出的关键词很多，我们同样可以设定阈值 x，即只处理优先级更高的那部分关键词（通常使用频率越低，优先级越高）。

Measure the search engine

如何评估一个搜索引擎？可以从以下几个方面。

How fast does it index

即建立索引的速度，通常用 Number of documents/hourNumber\ of\ documents/hourNumber of documents/hour 来评估

How fast dose it search

即搜索的延迟，注意在考虑延迟的时候要同时考虑 index size，小规模谁来都一样，大规模下跑得快才是真快

Expressiveness of query languge

翻译软件说这个叫“查询语言的表达能力”，嗯，怎么感觉怪怪的。总之，可以理解为搜索引擎理解用户需求的能力。比如同样是 “apple”，有的语境下指的是 “real apple”，有的语境下指的是 “Apple company”，这就很考验搜索引擎的理解力了。在比如一个意思我可以用多种方式表述，搜索引擎是否能在不同表述方式下给出准确的结果，这也能反映出搜索引擎的理解力。

User happiness

突然变得好现实

分为两点

Data Retrieval Performance：数据检索能力，包括准确度，反应时间和索引规模

Information Retrieval Performance：信息检索能力，指的是搜索结果反映关键词间相关性的能力，比如搜索 “red apple”，需要给出 "red apple"的结果，而不是 “red” 和 “apple” 的结果。

Measure the relevancy

话接上文，介绍一下相关性的评估方法

首先，我们要准备一系列的网页，然后对搜索引擎进行一系列的模拟提问，每个提问和每个网页间组成了一个 pair，它们要么相关，要么无关，统计每一对的测试结果，列出以下表格

Relevant Irrelevant
Retrieved RRR_RRR​ IRI_RIR​
Not Retrieved RNR_NRN​ INI_NIN​

由上表可以求出两个评估因素：

Precision：P=RR/(RR+IR)P=R_R/(R_R+I_R)P=RR​/(RR​+IR​)，即准确度

Recall：R=RR/(RR+RN)R=R_R/(R_R+R_N)R=RR​/(RR​+RN​)，召回率？中文名不太确定

大概知道什么情况 Precision 比较高，知道什么情况下 Recall 比较低就行（我说这句是互文，不接受反驳）
