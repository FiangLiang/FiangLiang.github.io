4.1 Introduction

Processor，也就是我们熟知的处理器，也叫 CPU，是计算机基本组成部分之一，决定了计算机的时钟时长和 CPI，而这两个都是评估计算机性能的关键因素

A Basic RISC-V Implementation

这一章我们将学习构建处理器的基本原则和技术，并尝试构建一个能支持 RISC-V 中基础指令运行的处理器

那何为基础呢？可分为三类：

  • 以 load 和 store 等为代表的 memory-reference 指令
  • 以 add 和 or 等为代表的 arithmetic-logical 指令
  • 以 beq 等为代表的conditional branch 指令

我们不会涉及乘法除法这种复杂指令，也不会涉及浮点数的运算指令，我们要学的只是上面三类中的最基础，最常见，最具代表性的指令，但通过这些基础的指令，我们也能清楚的了解如何去构建一个 Processor

如何去构建这个 Processor 呢？Processor 是为指令的执行服务的，因此，我们得先直到，一条指令执行的时候，Processor 的内部到底发生了什么

对于每条指令，其总要经过一下两步：

  1. 通过 PC 的值从内存中读取指令
  2. 根据指令的字段（Field）去读取指令所需的寄存器

这还没完，接下来，我们要根据指令的类型采取不同行动：

  • meomory-reference

    内存调用指令需要先调用 ALU 计算出所操作的内存的地址，再通过地址去找寻对应内存，以执行内存的读写操作

  • arithmetic-logical

    算术逻辑指令需要调用 ALU 进行运算，最后将计算结果保存写入对应的寄存器中

  • conditional branches

    条件分支指令需要通过 ALU 进行条件的判断，根据判断结果来决定下一条指令的内存地址，也就是决定 PC 的值

可以看到，每种的指令的执行都要用到 ALU，除此之外，指令执行过程中还涉及到了 PC，内存，以及寄存器等结构，它们共同作为 Processor 中不可或缺的部分发挥着作用，如下图

The ALU Control

这里的 ALU 只涉及 4 种基本操作：And, Or, add, sub

Why a Single-Cycle Implementation is not Used Today

标题为“为什么现在不使用单周期实现”，也就是说我们目前提到的 Processor 都是基于单周期实现的

单周期什么意思呢？也就是每条指令的运行周期相等，且均为 1 clock cycle。这样处理起来可能比较方便，但是实现却十分低效。究其原因，因为我们保证所有指令都是 1 clock cycle ，所以 clock period 的值就要取决于所有指令中耗时最长的那条，这样那些本来能很快运行的指令就被“拖累”了（多数时候 load 的耗时会高于其他指令的耗时）。

对小指令集而言，单周期设计或许可以接受，但如果要实现浮点单元或更复杂的指令集，单周期设计的表现将十分差劲

下一节我们将介绍“流水线”的技术，其使用与单周期类似的 datapath，但效率会显著优于单周期

4.5 An Overview of Pipelining

**流水线（Pipelining）**是一种能使多条指令重叠执行的实现技术。

以洗衣服为例，假如洗一次衣服可以分为一下 4 个阶段（stage）：

  1. 用洗衣机洗衣服
  2. 用烘干机把衣服烘干
  3. 折叠衣服
  4. 把衣服收起来

假如我们要洗很多次衣服，普通方法是每次洗衣服都按照这个流程跑一遍，整个过程结束以后再进行下一次洗衣服；而流水线的方法类似：洗衣机洗完，把洗好的衣服放入烘干机，同时立刻把下一次的衣服塞进洗衣机，也就是允许洗衣机和烘干机同时工作，后面两个步骤也是同理。通过下面的示意图可以看出两种方法的差异：

对于单个工作，流水线技术并没有缩短其运行时间；但是由于多个工作可以并行地执行，提高了系统的吞吐率（throughput），从而能在单位时间内完成更多的工作。

这一原则可以用于构造处理器，我们可以把 RISC-V 指令的执行分为以下 5 个阶段：

  1. 取指（IF）：从指令存储器中取出指令
  2. 译码（ID）：读取寄存器并译码指令
  3. 执行（EX）：执行操作或计算地址
  4. 访存（MEM）：访问数据存储器中的操作数（if necessary）
  5. 写回（WB）：将结果写入寄存器（if necessary）

如果各个阶段用时相等，理想状态下，流水线带来的加速比约等于阶段数

Designing Instructions Sets for Pipelining

RISC-V 在设计原则上也考虑到了流水线技术的运用，有很多对流水线友好的设计：

  • 所有 RISC-V 指令长度相等，这有利于指令的取出和译码
  • RISC-V 指令格式较少，而且源寄存器、目标寄存器的位置在各类指令中保持一致
  • RISC-V 内存操作只能通过 load 和 store ,且不会对存取的结果做进一步的运算，这样我们就可以利用 EX 的阶段去计算内存地址，在下一个阶段去访问内存，而不需要其他阶段

Pipeline Hazards

流水线设计中可能会遇到一些情况，使得在下一个时钟周期中下一条指令无法执行，这种情况叫做 “冒险（hazard）”，具体来说可分为以下三种情况：

Structural Hazard

结构冒险：硬件不支持多条指令在同一个时钟周期执行

由于 RISC-V 指令系统是面向流水线设计的，所以这类问题一般可以避免。

Data Hazard

数据冒险：一个步骤必须等待另一个步骤完成导致的流水线停顿

如果出现类似这样的指令序列

可以看到相邻的两条指令中调用了同一个寄存器，由于后面一条指令的执行需要依赖前一条指令的结果，所以我们只好被迫终止我们的流水线，这其实是非常浪费时间的。

一种解决数据冒险的方法叫做前递（forwarding）或旁路（bypassing），我们可以通过添加额外的硬件资源，提前从缓冲区中提取到后一步所需要的数据，而不需要等到前一条指令执行完毕

具体原理可以参照下图：

尽管前递的效果很好，但不能避免所有的流水线停顿，仅当目标阶段在时间上晚于源阶段时（可以参考上图），前递路径才有效。

这种问题常发生在**载入-使用型数据冒险（load-use data hazard）**中（之前也说过 load 指令耗时偏长），可以参考下图

这时流水线必须停顿一个阶段，在图中用 bubble 表示，**流水线停顿（pipeline stall）**是流水线的一个重要概念，之后的章节中我们会介绍如何解决这类冒险

Control Hazard

控制冒险：一个步骤如何执行需要等待另一个步骤的执行结果

这种问题主要出现在分支指令的处理上，解决方法是 “预测（prediction）”，假设一个预期结果，按照预期结果先执行之后的步骤。如果实际结果与预期不同，才会发生停顿。

4.6 Pipelined Datapath and Control

Datapath

在上一节提到，流水线设计把 RISC-V 指令的执行分为五个阶段：

  2. ID：Instruction decode and register file read

Control

datapath 大致就这样了，接下来是 control 的设计，这部分可以借鉴单周期时钟中的设计

流水线设计的 datapath 与原单周期时钟设计的 datapath 相比最显著的不同就是在每个阶段之间多了 4 个流水线寄存器，但是这写流水线寄存器在每个时钟周期都被写入，因此不需要额外的控制信号。

为了有条理的说明我们在 datapath 上添加的 control ，我们依然是将控制线按照阶段分为 5 组：

  1. 取指：涉及读指令寄存器和 PC，但它们的控制信号总是有效的，故不需要额外控制信号
  2. 指令译码/读寄存器堆：由于指令中寄存器的位置固定，故也不需要额外的控制信号
  3. 执行/地址计算：涉及到 ALU ，要设置的信号为 ALUOp 和 ALUSrc，前者负责选择 ALU 的操作，后者负责选择 ALU 的输入
  4. 存储器访问：本阶段要设置的信号为 Branch, MemRead, MemWrite。这些信号分别负责分支、加载和存储指令。
  5. 写回：两条控制线，分别为 MemtoReg 和 RegWrite，前者负责选择待存储的值，后者控制写入行为

如何实现“停顿”这个行为呢？具体涉及下面三个点：

  • 停止 IF：为了保持指令不变，不能改变 PC 寄存器的值，要为 PC 寄存器添加写信号 PCWrite，当 PCWrite = 0 时，就能做到停止 IF
  • 停止 ID：不能改变 IF/ID 流水线寄存器的值，要为该寄存器添加写信号 IF/IDWrite，当 IF/IDWrite=0 时，我们停止对 IF/ID 流水线寄存器的写入
  • 停顿时，虽然 CPU 仍在运行，但实际上没有改变如何状态，这种情况称为 “空操作（nops）”。而为了保证所有元件状态不变，还要确保所有控制信号均为 0

4.8 Control Hazards

上一节介绍了数据冒险，这一节介绍的是发生在分支条件的冒险，也就是之前提到的控制冒险。

以 beq 为例，它需要在 ID 阶段结束后才能知道跳转地址，在 MEM 阶段结束后才能决定是否跳转，这一过程如下所示：

可以看到，在 MEM 阶段结束后，CPU 才能确定 beq 指令需要跳转，但此时已经多执行了 3 条指令，因此需要将这 3 条指令从 CPU 中全部清除

我们之前提过，解决 Control Hazards，可以采用 “预测” 的方法，下面我们介绍两种 “预测” 的手段

Static Branch Prediction

静态分支预测，原理可以概括为 “always not taken”

对一个条件分支指令，我们总假设它 “不发生跳转”，如果假设失败，我们只需要将正在运行的指令抛弃即可

抛弃指令的具体做法：将控制信号置 0，同时清除前 3 个阶段的指令，因为此时分支指令正在 MEM 阶段

然而这种做法成本过高，为了降低成本，我们可以减少需要清除的指令数。我们需要将条件分支提到前面来，具体来说需要做到这两件事：

  • 提前计算分支跳转指令

    我们将分支地址的计算移到 ID 阶段，可以在 ID 阶段添加一个专门的分支加法器。任何指令执行到 ID 阶段都会进行加法运算，但是只有在条件分支指令时糊用到它的计算结果

  • 提前进行分支跳转决策

    仍然是在 ID 阶段，我们增加一个相等检验单元，用于比较两个寄存器的值，其判断结果会存在 ID/EX 寄存器中。

    提前进行寄存器值的比较，可能会同时产生数据冒险，所以我们还需要额外的前递和冒险检测装置。

如此一来，跳转分支可以在 ID 阶段就完成，因此我们只需要清除前面 IF 阶段的一条指令即可。每当发现有一个需要跳转的 beq 指令，我们通过 IF.Flush 控制信号来实现“清除”操作，同时使此时的 ID 执行空操作

由于条件分支指令的寄存器的调用提前到了 ID 阶段，所以前递操作会产生更多的停顿，如果条件分支指令前有一条 ALU 指令，则条件分支指令需要停顿一个周期；如果前面是一条 load 指令，则需要停顿两个周期

Dynamic Branch Prediction

在更复杂的流水线中，静态分支预测正确率不够高，的成本还是太大，于是我们引入了动态分支预测——在程序执行过程中，根据上一条条件分支指令的运行结果来预测分支是否跳转。

实现动态分支预测需要引入分支预测缓存（branch prediction buffer），或者称为分支历史表（branch history table），本质上就是一块很小的内存

最简单的分支预测缓存只有 1bit，也就是存储上一次分支是否发生跳转，并据此预测下一次分支执行的情况，缺点是内侧循环分支会有 2 次预测错误，第一次迭代和最后一次迭代

2 bit 缓存能够提高预测精度，只有在连续两次预测错误时再改变预测，这个过程可以用下面的状态机表示：
